# Agents that continue where they left off — at 23 % of the cost

## What we measured on Terminal-Bench 2.1, and what we're building next

*Published 2026-06-02*

Most AI agents today are amnesiacs. They restart from zero every session. The lesson your coding agent learned the hard way last week is gone today. Every developer on your team rediscovers the same gotchas. Every project hits the same QEMU + linker landmines fresh. Lessons never compound.

There's no axis on any agent benchmark leaderboard that measures whether an agent *gets better* the more you use it.

There's also no axis that measures what it costs to run them. That one's missing too.

We spent two weeks building and measuring an open-weight agent stack that fixes both gaps. On Terminal-Bench 2.1, a learned skill file — generated by reflecting on one prior failure trajectory and loaded by the next session at run time — lifts an open-weight model to **81 % of frontier accuracy at 23 % of the per-token cost**. The mechanism *is* the marketing: each session ends with the next session's bootstrap data, and the agent quietly improves at your stack every time you use it.

This post is the first measurement that the loop works on a hard benchmark. The rest of this page explains what we measured, how the loop is built, and what comes next.

## What we measured

89 tasks. K=3 attempts per task. TB-standard aggregated score (per-task mean of the first K attempts, then averaged across tasks). 87 of 89 tasks measured under every condition; two errored everywhere (we report them with an asterisk rather than dropping them silently).

Every number below is regenerated by a single deterministic script — [`scripts/aggregate.py`](scripts/aggregate.py) — that reads [`data/raw-trials.json`](data/raw-trials.json) (699 per-trial records, ~230 KB) and writes the canonical results. Clone the repo, run the script, reproduce every number on this page exactly. In about two seconds.

|---|---|---|---|---|
| Claude Opus 4.8 ([TB-2.1 leaderboard #2](https://www.tbench.ai/leaderboard/terminal-bench/2.1)) | **78.9 %** | — *(not published)* | — | 1× (frontier reference) |

The hypothetical Opus 4.8 cost ($1.64 per attempt) comes from applying [OpenRouter list pricing](https://openrouter.ai/anthropic/claude-opus-4.8) ($5/M input, $25/M output) to the actual token usage we measured across 214 trials — *if you ran our same prompts on Opus 4.8, that's what it would cost.* The 4.4× ratio is conservative; with output-heavy reasoning workloads, the multiple grows.

The honest read: we don't match frontier accuracy. We land at 81 % of it (64.0 / 78.9) at 23 % of the cost. That's a price-quality cell the leaderboard doesn't capture and almost nobody publishes data on. For workloads where 78.9 % is overkill and dollar cost matters — most of the workloads our friends are actually running — that cell is open.

But the 4.6-point lift from baseline to C-skill is the more important number on this page. Not because it's large in absolute terms — it isn't — but because it's the first signal that an agent stack can *continue from where it left off* on a benchmark this hard. The cost number gets people's attention. The mechanism that produced the lift is what we're actually building.

## How the agent improves between sessions

The skill is a single Markdown file. The agent reads it from its filesystem before starting a task. No model weights are touched. No fine-tuning, no RL.

The loop has two stages:

1. A **Reflector** call reads one prior failure trajectory — the verbatim record of what the agent tried, what it observed, where it gave up. It extracts the single load-bearing mistake. Not a list of issues — just the one thing that, if the agent had known it going in, would have fixed the run.
2. A **SkillCreator** call writes the actual SKILL.md file. Frontmatter (name + a "when this triggers" description), a Principle section (1–2 sentences on the conceptual gap), a Procedure section (up to 5 numbered steps with concrete commands), an optional Pitfalls list.

One trajectory in, one ~80-line skill out. The next session loads the file at run time and follows the procedure. The agent doesn't *remember* — the skill library does. And the library grows with every failure.

That's the continuum. Every session ends with the next session's bootstrap data. Skills accumulate. The same skill works for any model on the same task — so a skill authored from a cheap open-weight failure helps a frontier model too. Skills authored on one developer's machine help every other developer. Skills accumulated on one project transfer to similar projects. The thesis is that this multi-axis continuity, not raw model capability, is what makes agents quietly useful over time.

We deliberately ran the simplest version of the loop first: one trajectory in, one skill out, no iterative refinement, no validation gate. So we'd know how much lift the basics alone produce. That's the +4.6-point number. Adding a validation gate ([SkillOpt-style](https://microsoft.github.io/SkillOpt/)) is the next obvious win and is on the roadmap.

## Two author models, tested honestly

The execution model is GLM-5.1 in every cell. The only thing that varies is who *authored* the SKILL.md file. We tested:

- **B**: GLM-5.1 authors its own skills (the cheap open-weight self-author)
- **C**: Claude Opus 4.6 authors the skills (the stronger separate author)

The lift compared to the unaided GLM-5.1 baseline:

- **B**: +3.4 pp (59.4 % → 62.8 %)
- **C**: +4.6 pp (59.4 % → 64.0 %)

The GLM-authored skill captures **about three-quarters of the lift** the Opus-authored skill provides. That matters: you don't need a frontier author to make this work. You can author and execute with the same cheap open model and still get most of the value. The skill is the asset, not the author.

## What kind of lessons the skills actually encode

The skills aren't generic advice. They're mechanical, specific, transferable.

One skill identified a default tool behavior that was silently destroying input data — five lines of "back this up before you touch the live file" turned a task that had never passed into one that passed reliably. Another encoded a library-API subtlety that no model in our test set figured out unaided — a one-paragraph SKILL.md took the task from 0/3 attempts passing to 3/3. A third correctly diagnosed an obscure environment-level interaction crashing a build chain (a competing skill from a different author misdiagnosed it and actively regressed the task — a candid reminder that we don't have a validation gate yet, and shipping every skill the author produces means sometimes shipping skills that hurt).

Detailed case studies — full SKILL.md text, the failure trajectories that produced them, side-by-side comparisons of what each author got right and wrong — are coming in dedicated follow-up posts. The aggregate above is what these specific wins add up to across 89 tasks.

## What we noticed along the way

About halfway through the run, we hit something uncomfortable.

The agent harness defaults to a 30-minute per-task budget. When we triaged failures we'd flagged as capability gaps, we noticed several of them looked suspicious — clean runs that ended exactly at the timeout, no obvious confused state. So we re-ran 23 of those at a 60-minute budget instead.

**15 of 23 (65 %) solved on the first try.** They weren't capability failures. They were budget-starved.

That single fix moved our measured open-weight baseline from 53.8 % to 59.4 % — and meant most of what we'd initially attributed to skill lift was actually a measurement artifact. The aggregator was correctly summarizing wrong inputs.

The methodological implication is uncomfortable: if you publish or trust agent benchmark numbers, your default timeout probably correlates with model speed. Slower-but-more-thoughtful models look worse than they are. Faster models look better. Any "improvement" you measure on top of a mis-budgeted baseline inherits that bias. If you're running an iterative skill-optimization loop with a validation gate (SkillOpt, GBrain), the gate is comparing two equally-corrupted numbers and "accepted edits" become statistical noise.

A dedicated post on this single finding is coming next week — it deserves its own thread.

## Why "ContinuumAI"?

Every other axis on which AI is improving is discrete: a bigger model, a longer context window, a smarter sampling strategy. Releases jump. The user upgrades. The old version is discarded.

The continuum is different. It runs along a single axis — *time and repetition* — and improves smoothly. Yesterday's failure becomes today's skill. This week's pattern becomes next quarter's procedural knowledge. There's no version jump because every session is an increment.

We think the next decade of useful agents looks more like that continuum than like the next model release. Not because models will stop getting better — they obviously won't — but because the cost-quality gap between *the best frontier model* and *a competent open-weight model + an accumulated skill library* is going to keep closing from below. And the workloads in your inbox right now are not gated on the frontier. They're gated on *getting better at your specific stack the more they run.* That's continuity. That's what we're building.
