# 81 % of frontier accuracy at 23 % of the cost

## What an open-weight agent with a learned skill actually delivers on Terminal-Bench 2.1

*Published 2026-06-02*

Every agent benchmark leaderboard publishes accuracy. None of them publish cost.

So when you look at [Terminal-Bench 2.1](https://www.tbench.ai/leaderboard/terminal-bench/2.1) — currently topped by Codex CLI on GPT-5.5 at 83.4 % and Claude Code on Opus 4.8 at 78.9 % — you can see how good those agents are. What you can't see is how much it costs to run them. And for anyone actually shipping agents into production, that's the most important number missing from the table.

We spent two weeks measuring it. Specifically: what does an open-weight stack actually deliver on the same benchmark — same task hardness, same harness, real dollar costs attached?

**Result: GLM-5.1 with a learned skill file hits 81 % of frontier accuracy at 23 % of the per-token cost.** Here's how it works, what we measured, and what we found along the way.

## What we measured

89 tasks. K=3 attempts per task. TB-standard aggregated score (per-task mean over the first K attempts, then averaged across tasks). 87 of 89 tasks measured under every condition; two errored everywhere (`extract-moves-from-video`, `path-tracing`). Every number below is regenerated by a single deterministic script — [`scripts/aggregate.py`](scripts/aggregate.py) — that reads [`data/raw-trials.json`](data/raw-trials.json) (~700 trials, ~230 KB) and writes [`data/canonical-results.json`](data/canonical-results.json). Clone the repo, run the script, reproduce every number on this page exactly.

| Stack | TB-2.1 pass rate | $ / attempt | $ / passed task | Per-token cost vs Opus 4.8 |
|---|---|---|---|---|
| Claude Opus 4.8 (TB-2.1 leaderboard #2) | **78.9 %** | — *(not published)* | — | 1× (frontier reference) |
| GLM-5.1 baseline, no skill | 59.4 % | $0.36 | $0.61 | **0.23×** |
| GLM-5.1 + GLM-authored skill | 62.8 % | $0.39 | $0.62 | **0.23×** |
| **GLM-5.1 + Opus-authored skill** | **64.0 %** | **$0.37** | **$0.58** | **0.23×** |

The hypothetical Opus 4.8 cost ($1.64/attempt) comes from applying OpenRouter list pricing ($5/M input, $25/M output) to the actual token usage we measured across 214 trials — *if you ran our same prompts on Opus 4.8, that's what it would cost.* The 4.4× ratio is the conservative number; with output-heavy reasoning workloads, the multiple grows.

The honest read: we don't match frontier accuracy. We land at 81 % of it (64.0 / 78.9) at 23 % of the cost. That's a price-quality cell the leaderboard doesn't capture and almost nobody publishes data on. For workloads where 78.9 % is overkill and dollar cost matters — most of the workloads our friends are actually running — that cell is open.

## How the skill works

The skill is a single Markdown file. The agent reads it from `/tmp/hermes/skills/` before starting a task. No model weights are touched. No fine-tuning, no RL.

We borrowed the two-stage authoring pipeline from [Letta's skill-learning recipe](https://www.letta.com/blog/skill-learning):

1. A **Reflector** call reads one failure trajectory — the verbatim record of what the agent tried, what it observed, where it gave up. It extracts the single load-bearing mistake. Not a list of issues — just the one thing that, if the agent had known it going in, would have fixed the run.
2. A **SkillCreator** call writes the actual SKILL.md file. Frontmatter (name + a "when this triggers" description), a Principle section (1–2 sentences on the conceptual gap), a Procedure section (up to 5 numbered steps with concrete commands), an optional Pitfalls list.

One trajectory in, one ~80-line skill out. We considered iterative refinement, validation gates, and edit-budget tricks à la [SkillOpt](https://microsoft.github.io/SkillOpt/). We deliberately ran the simplest version that works first, so we'd know how much lift the basics alone produce.

## Two author models, tested honestly

We tested two authors for the SKILL.md files:

- **B**: GLM-5.1 authors its own skills (the cheap open-weight self-author)
- **C**: Claude Opus 4.6 authors the skills (the stronger separate author)

The execution model in both cases is GLM-5.1, with the skill file mounted at runtime. The only thing that varies is who wrote the skill.

The lift compared to the unaided GLM-5.1 baseline:

- **B**: +3.4 pp (59.4 % → 62.8 %)
- **C**: +4.6 pp (59.4 % → 64.0 %)

The GLM-authored skill captures **about three-quarters of the lift** the Opus-authored skill provides. That matters: you don't need a frontier author to make this work. You can author and execute with the same cheap open model and still get most of the value. The skill is the asset, not the author.

The +4.6 pp Opus-authored number is in the same range as Letta's published TB-2.0 trajectory-only lift (+9 pp on the easier predecessor benchmark, without a validation gate). On the harder benchmark, with no validation gate yet, we'd expect to land where we did — and we did.

## Three concrete wins

Aggregate numbers are useful for sizing the bet. Concrete examples are what convince you something is real. Three tasks where a single skill moved the agent from `pass^3 = 0` (never reliably solved across three attempts) to `pass^3 = 1` (all three pass cleanly):

**`db-wal-recovery`** — Reconstruct data from a SQLite database where the Write-Ahead Log file is corrupted. The agent, every time, opened the database directly with `sqlite3`. That command silently processes the WAL and *discards it on close if it can't be parsed* — destroying the data the task asks you to recover. The skill encoded the single load-bearing insight: back up the WAL bytes before any `sqlite3` invocation, then analyze the backup, not the live file. Result: 0/3 → 3/3.

**`mteb-retrieve`** — Use the `mteb` library to load a BGE embedding model and rank documents by similarity. The agent kept reaching for `sentence_transformers.SentenceTransformer` directly, which skipped the model-specific query-prefix preprocessing BGE expects (a literal Chinese instruction the model uses internally). Without the prefix, embeddings come out wrong, rankings come out wrong. The skill: use the loader the task mandates, even when the underlying library seems easier. Result: 0/3 → 3/3.

**`polyglot-rust-c`** — Write a single source file that compiles and runs as both Rust and C. The agent's builds kept crashing with `SIGSEGV` during linking. The actual root cause was a jemalloc + QEMU + `cc` interaction crashing the linker — invisible unless you knew to look for the `<jemalloc>: MADV_DONTNEED does not work` line in stderr. The Opus-authored skill identified this directly and gave a one-line environment-variable fix. The GLM-authored skill misdiagnosed it as a command-line flag issue, and actually *regressed* the task from 2/3 → 1/3. Result with C-skill: 2/3 → 3/3. Result with B-skill: a candid reminder that we don't have a validation gate yet, and shipping every skill the author produces means sometimes shipping skills that hurt. (Adding the gate is next on the roadmap.)

These are mechanical, transferable lessons. The same SKILL.md file works on any model that runs the same task. A frontier model would also benefit from them; we just don't need a frontier model to *author* them.

## What we noticed along the way

About halfway through the run, we hit something uncomfortable.

The Hermes agent harness defaults to a 30-minute per-task budget. When we triaged failures we'd flagged as capability gaps, we noticed several of them looked suspicious — clean runs that ended exactly at the timeout, no obvious confused state. So we re-ran 23 of those at a 60-minute budget instead.

**15 of 23 (65 %) solved on the first try.** They weren't capability failures. They were budget-starved.

That single fix moved our measured open-weight baseline from 53.8 % to 59.4 % — and meant most of what we'd initially attributed to skill lift was actually a measurement artifact. The aggregator we'd been using was correctly summarizing wrong inputs.

The methodological implication is uncomfortable: if you publish or trust agent benchmark numbers, your default timeout probably correlates with model speed. Slower-but-more-thoughtful models look worse than they are. Faster models look better. Any "improvement" you measure on top of a mis-budgeted baseline inherits that bias. If you're running an iterative skill-optimization loop with a validation gate ([SkillOpt](https://microsoft.github.io/SkillOpt/), [GBrain](https://github.com/microsoft/SkillOpt)), the gate is comparing two equally-corrupted numbers, and "accepted edits" are statistical noise.

Practical recommendation: before you publish or trust any agent-benchmark number, run a cheap recovery pass at 2× budget on every "failed" task. It's one extra K=1 attempt per failure, a few dollars of compute, and it can completely change what you think you're measuring. The script we used is [in the repo](https://github.com/mrazakhan/LongLearningAgents/blob/main/scripts/eval_glm_vanilla_long_budget_recovery.py).

A dedicated post on just this finding is coming next week — it deserves its own thread.

## Why "ContinuumAI"?

Most AI agents today are amnesiacs. You give a coding agent a task. It figures something out, makes mistakes, eventually finishes (or doesn't). Next time you give it a similar task, it starts from zero. The lesson it learned the hard way last week is gone. Every session starts from scratch — every developer in the team rediscovers the same gotchas, every project hits the same QEMU-jemalloc-linker landmine fresh.

That's the gap ContinuumAI is built to fill: agents whose sessions form a *continuum* of skills. Each failure leaves behind a transferable lesson the next session loads at run time. The agent gets quietly better at your stack the more you use it — not by retraining, not by storing more chat history, but by accumulating a curated, audit-able library of short, named, mechanical lessons.

The TB-2.1 result above is the first measurement that this approach works on a hard benchmark, with an open-weight executor, at fractional cost. It's the start of the continuum, not the end of it.
